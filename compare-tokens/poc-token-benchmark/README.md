# Claude Code Token Benchmark POC

Compares token consumption of the **same `CLAUDE.md`** across different
operating systems and file encodings.

## What is measured

| Variable | Values |
|---|---|
| **Environment** | macOS · Linux · WSL2 · Windows native |
| **Line endings** | `LF` vs `CRLF` |
| **BOM** | UTF-8 BOM present vs absent |
| **Tokens** | input · output · cache creation · cache read |
| **Time** | wall-clock duration per run |

The prompt is fixed and minimal (`Respond with exactly one word: READY`)
so output tokens are constant and don't influence the comparison.

## Prerequisites

### All environments
- **Claude Code** installed and authenticated (`claude --version` works)
- **Python 3.9+** in PATH (`python3 --version` or `python --version`)
- **Node.js** (for `npx ccusage` if you want a second opinion — optional)

### macOS / Linux / WSL2 (bash)
```bash
which claude   # must return a path
which python3  # must return a path
```

### Windows native (PowerShell)
```powershell
claude --version   # must work (Git Bash must be in PATH or claude.exe found)
python --version   # python or python3
```

> **Note:** Claude Code on Windows requires Git Bash. The official installer
> puts the binary at `%USERPROFILE%\.local\bin\claude`. Make sure that path
> is in your `PATH` environment variable.

```bash

git clone <repo-url>
```

## Cross-environment comparison (recommended workflow)

To compare macOS vs WSL2 vs Windows with the same variants:

1. Run all 4 variants on **macOS** → results saved as `macos_lf_*.json`, etc.
2. Copy the whole project to **WSL2** (or run from a shared path):
   ```bash
   # In WSL2, pointing to the same results/ dir
   bash scripts/run_test.sh lf
   bash scripts/run_test.sh crlf
   bash scripts/run_test.sh lf_bom
   bash scripts/run_test.sh crlf_bom
   ```
3. Run on **Windows native** (PowerShell), same results/ dir.
4. Run `analyze_results.py` once — it reads ALL JSONs and compares them together.

> **Tip:** If WSL2 and Windows share a path (e.g., `/mnt/c/Users/.../poc-token-benchmark`),
> you can write results from all environments to the same `results/` folder.

## Encoding variants explained

| Variant | Line ending | BOM | Notes |
|---|---|---|---|
| `lf` | `\n` | no | Unix default, Git default, **baseline** |
| `crlf` | `\r\n` | no | Windows default (Notepad, etc.) |
| `lf_bom` | `\n` | yes | UTF-8 BOM — some Windows tools add this |
| `crlf_bom` | `\r\n` | yes | Full Windows encoding |

The BOM (`\xEF\xBB\xBF`) adds **3 bytes** to the file. Whether the tokenizer
treats it as tokens or strips it is part of what this POC measures.

CRLF adds **one extra byte per line** compared to LF. A file with 50 lines has
50 extra bytes in CRLF format. Whether those `\r` characters become extra tokens
is the key question.

## Troubleshooting

### `claude: command not found`
- macOS/Linux: `export PATH="$HOME/.local/bin:$PATH"` then retry.
- Windows: Make sure `%USERPROFILE%\.local\bin` is in your PATH.
- WSL2: Install Claude Code **inside WSL2** separately from the Windows install.

### `No session file found after the given timestamp`
Claude Code may write the session file slightly after the process exits.
Wait 2-3 seconds and re-run `parse_session.py --after <ts>` manually.

### Token counts are all zero
The JSONL format may vary by Claude Code version. Open the session file
(`session_file` field in the result JSON) and inspect it manually. Look for
a `usage` key in any line.

### Results differ between identical runs
Claude Code uses **prompt caching**. The first run on a fresh session creates
cache (`cache_creation_tokens > 0`). A second run on the same content will
show `cache_read_tokens > 0` instead. This is expected and interesting data.
